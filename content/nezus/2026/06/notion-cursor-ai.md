## Notion 把代码写交给了 Cursor ：一个文件夹就是一个 AI 员工

凌晨两点，产品经理在 Notion 文档里@了一下 Cursor。

三分钟后，一个完整的功能模块已经躺在 PR 里，等着 review。

这不是科幻场景。这是 2025 年的真实工作方式。


## 当文档编辑器开始"雇佣"AI

Notion 做了一件大胆的事：**把编码能力整个外包给了 Cursor**。

不是简单的 API 对接，不是插件集成，而是通过 Cursor SDK ，把完整的智能体运行时嵌入了自己的产品里。

这意味着什么？

意味着你在 Notion 里敲下@Cursor 的那一刻，你不是在调用一个工具——你是在**指派一个员工**。


## 讨论串 = 工位，消息 = 任务

Notion 重新定义了协作的基本单元。

**一个讨论串，对应一个 Cursor 智能体。一条消息，就是一次完整的任务执行。**

你在数据库里建一条记录，写上"需要用户登录页面的 A/B 测试"。

Cursor 智能体自动接单。

它开始规划技术方案，写代码，跑测试，验证功能。做完之后，直接创建一个 Pull Request。

整个过程，**没有人工干预，没有中间商**。


## 断连？不存在的

最让人惊喜的是技术细节。

结果通过 SSE 流式传输，**支持断连恢复**。

你可以关掉浏览器去喝咖啡，回来时进度条还在那里，智能体还在干活。

这不是 demo 级别的玩具，这是生产级别的可靠性。


## Cursor SDK ：不只是模型

很多人以为 AI 编码就是调用大模型 API。

**大错特错。**

Cursor SDK 提供的是与生产环境完全相同的能力：

- 同样的模型
- 同样的运行时
- 同样的远程 MCP 支持

这不是"Cursor 的简化版"，这是**Cursor 本体**。

Notion 用户拿到的，和 Cursor Pro 用户用的是同一个东西。


## 数周，不是数月

最恐怖的是时间线。

**Notion 通过 Cursor SDK ，在数周内完成了编码智能体的产品化集成。**

不是数月，不是半年，是几周。

这背后是 SDK 的成熟度，也是两家公司对"AI 原生工作流"的共同理解。


## 文件夹时代的终结？

传统软件开发，一个文件夹对应一个模块。

现在，**一个文件夹对应一个 AI 员工**。

Notion 和 Cursor 正在重新定义"团队协作"的边界。当 AI 可以独立接单、交付、验证，人类的角色是什么？

是需求定义者，是质量把关者，是战略决策者。

**执行，正在成为 AI 的工作。**
