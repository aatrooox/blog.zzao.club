非开发者在使用 AI 辅助开发（**其实是你在辅助 AI**）自己的产品时

难免会对各种技术栈、各种 skill、插件配置、提示词等感到头疼，或是选择困难，并且很多 SKILL 你不用是不知道它好不好用

但是全部问 AI ，一是无法准确描述自己都不熟悉的领域，二是 AI 更拿不准你也想不明白的事

所以这篇文章**不会让你做任何选择，直接分享一套可以做出一个项目的工具链**。**目的是带你看一下整个开发流程的全貌**，你可以根据这套工具链反问 AI ，还有哪些类似工具，从而补全自己的知识图谱。

**后续文章**也会基于这套工具链从零完成一个完整的项目并部署上线。

## 零、你的目的是什么

如果是为了学习，你应该在发现有不理解的地方时，马上去询问它的运作机制，在脑海中构建细节，然后在实践中反复纠正。

如果是为了产品上线，你应该更多的关注产品上线后的推广工作，因为写代码这件事 AI 已经做的很好了， SKILL 也有别的开发者替你完善好了。先推广后开发都是来得及的，唯一纠结的点可能在 UI/UX 上，非常考验个人审美以及能否表达自己的审美标准。

想清楚目的，可以避免在海量的信息中找到自己的关注点，其余的哪怕不懂其实也毫无影响，避免过度焦虑～

## 一、全局提示词

基于以下四个原则，约束和规范 AI 行为。

提示词的这些约束是 AI 在编码时经常会出现的问题，所以非常适合放在全局使用。

| **原则**     | **解决什么问题**      |
| :--------- | :-------------- |
| **编码前思考**  | 错误假设、隐藏困惑、缺少权衡  |
| **简洁优先**   | 过度复杂、臃肿抽象       |
| **精准修改**   | 无关编辑、触碰不应碰的代码   |
| **目标驱动执行** | 通过测试优先、可验证的成功标准 |

推荐使用 **Karpathy** 的四大原则来封装的 skill ：<https://github.com/multica-ai/andrej-karpathy-skills>

1️⃣ 在全局/本项目安装，特点是每次会话都加载

```Shell
## 在全局 ~/.claude/CLAUDE.md 每个项目都会加载
## 在项目级 .claude 放置，则会只在当前项目下启动 cc 时加载
curl -o CLAUDE.md https://raw.githubusercontent.com/forrestchang/andrej-karpathy-skills/main/CLAUDE.md
```

2️⃣ 官方推荐的 Claude Code 中安装方式：

```Shell
/plugin marketplace add forrestchang/andrej-karpathy-skills
/plugin install andrej-karpathy-skills@karpathy-skills
```

注意，这种方式不会让此提示词每次会话都加载。只会根据描述自行判断加载时机，所以我还是推荐用第 1️⃣种方式。

## 二、superpowers

**[superpowers](https://github.com/obra/superpowers)** 是一组 skill 的集合，是一套软件开发方法论。（安装方式可以去 github 查看）

**brainstorming** — 头脑风暴与需求设计。把一个模糊的想法通过对话式协作打磨成完整的设计文档（ spec ）。适合项目初期"我有个大概想法但还不清楚怎么做"的阶段。

**writing-plans** — 编写实施计划。基于 brainstorming 产出的 spec ，将其拆解为具体的、可执行的多步计划，每步都有明确路径和代码改动点，强制"零占位符"原则，不允许写"TODO: 以后补充"这种东西。

**executing-plans** — 执行计划。按照 writing-plans 产出的计划逐步实施，每个步骤完成后有 review 检查点，确保不跑偏。

这三个是核心流水线：**brainstorming → writing-plans → executing-plans**，从模糊想法到可执行步骤再到落地。

## 三、高保真原型

推荐用宝玉老师的 [baoyu-design](https://github.com/JimLiu/baoyu-design) 来做高保真原型，它把 Claude Design 打包成一个可移植的 **Agent Skill （技能）**

可以生成：精致的 UI 稿、可交互原型、线框图、落地页、仪表盘、移动 App、幻灯片 —— 全部以「自包含 HTML」的形式产出。

产物会落到 `designs/<项目名>/` 目录下，是可以纳入版本控制、二次分叉、导出或直接上线的自包含 HTML。

并且会帮你启动一个本地实时预览，**在实时预览里点选任意元素，说出想改成什么** —— Agent 就会去改对应源码，完成一次快速、可视化的二次编辑。

## 四、从需求到产品雏形

**但是！**

**大部分情况下，产品的构思是在脑子里有一部分，但是 UI 长啥样可能没有想法，一切都要靠 AI 来完成。**

这种情况下，我比较推荐，先用 **brainstorming** 把产品想法写出来，然后去「原型设计」里用 baoyu-design 出 wireframe （线框图）

**越模糊的东西越要先具象化，越具体的东西越往后放**。

不然模糊的东西会一直“不是自己想要的感觉”，然后反复修改计划

有了 wireframe ，经过多次迭代，是自己想要的感觉后，升级为 hi-fi design （也是 baoyu-design 里的）此时产品雏形就基本定下了

然后再去用 **writing-plans** 写实施计划，计划写完它会自动询问你有没有问题，要不要进行下一步。

所以整个流程可以是：**brainstorming → wireframe → hi-fi → writing-plans → executing-plans** 

## 五、部署

**我用 AI 写了一个超酷的 XXX ，快来看看吧： http\://localhost:3000 🤡**

当你把项目开发完成后，必须要放在一个可访问的服务器上才能被别人看到啊啊啊啊，所以必须要部署上线。

可以按照以下思路，构建自己的一个专用 skill ：

1️⃣  项目分析：扫描自己的项目目录，看 `package.json`、`requirements.txt` 这些标志文件，判断你用的是什么技术栈、有没有数据库、是全栈还是纯前端

2️⃣ 选平台：根据分析结果，自动推荐最合适的部署平台。纯静态就推 Vercel/Cloudflare （免费），核心原则是**能用托管就不碰 VPS。**此时你有自己的服务器最好，可以直接用 ssh config 直接配好自己服务器的秘钥，然后让 AI 给你定制部署脚本。

3️⃣ 执行部署：先验证本地构建通过，然后按对应平台的指南走部署流程，严格检查环境变量有没有泄露或者没同步到正式环境

4️⃣ 上线验证：输出一个 checklist ， HTTPS 有没有生效、API 通不通、数据库连不连得上、有没有错误日志等。

5️⃣ 提醒：告知你该监控用什么、日志去哪看、备份怎么做。

直接把以上信息告诉 AI ，应该可以帮你部署上线了。

然后你再让它把部署流程变成脚本或 SKILL ，方便下次执行就可以了。

## 总结

每一个环节有相当多的替代方案，比如 UI 可以用 Google Stitch 直接出 UI ，再导出到 Figma ，借助 MCP 使用。或者用 [ui-skills](https://www.ui-skills.com/) 里的 ui-ux-pro-max skill。

但产品一旦开发完，定型了，估计很少有人会去一直重构它。

所以只要有一种自己觉得满意的工具链就可以了。
