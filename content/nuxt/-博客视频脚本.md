## 介绍 Vue 和 Nuxt 区别

前端是使用HTML/CSS/JavaScript 这三种语言编写，部署到服务器上时也是这三种，访问了时服务器给了你一个 html 文件，这个文件里引入了 css 和 js，于是就有了样式和功能，就这么简单。

作为一个现代框架，比如 Vue或 React，不管它使用了什么 JS语法，定义了什么使用规则，最终的产物还是这三种。

我觉得了解这点还是比较重要的。

因为很多同学呢，直接从 Vue2Vue3开始学起，这时候 Webpack Vite 的生态已经很成熟很丰富了，你要想运行一个前端项目，要先安装 node 环境，然后下载依赖，然后在启动，开发的时候还要读很多繁琐的配置，甚至 HR 面试的时候还要问你会不会调优，最后还要打包，打出一堆看不懂的东西。这一堆东西加持起来，让人一看就很复杂，就好像前端已经不是以前的三大件了一样。

其实对于前端的产物，也就是实际部署到服务器上的东西来说，还是那三样东西。

而现代的前端框架可以看做是对前端做出了一些基建和约束。

什么是基建呢，就类似咱们国家的基建一样，要想富先修路。

框架对于前端来说就是修路，有了框架之后对于开发者完整的开发完一个产品提供了很大的便利，像是：脚手架启动, UI模板，表单校验，权限校验，管理系统，支付收款等等。都会围绕框架和插件展开，而且依靠开源社区，你可以找到几乎所有你想要的包。

所以我们只需要在脑子构思好自己要做个什么，然后开始去做，每次碰到问题，去对应的解决问题就行了。包括现在 AI 对开发的帮助是非常大的，以前你可能还需要去问问同事，问问网上的大神，现在的 AI 能提供给你的帮助，绝对强于你 80% 的同事。

不然你这一路上什么都自己敲的话是很麻烦的事。

那什么是约束呢，我个人认为呢，约束相对于团队/公司合作开发来说的，只是这个形式体现在了技术上。

什么意思呢，举个例子，你作为一个老师或者发起人要组织一些高中生参加一个户外活动，这时候你要说一些规则，比如说大家统一装束，统一时间，一起前往某个户外公园，到达后要点名/要限制活动范围。 这些规则目的是什么呀？ 作为发起人，你肯定不会想着一会去了先去玩个什么游戏吧，肯定是为了让活动圆满结束，尽量让所有人不发生意外。

那对于 Vue/React, 或是用 TS，都是一样的道理。对于公司来说，让产品顺利落地，对组长/leader 来说让大家编码风格统一，具备一定的扩展性，让这个项目代码能让另一个人快速接手，这是最重要的，所以要约束：格式化的规则, 文件名命名规则，TS必须写类型不能用 any 等等。

所以框架给了你便利，但也一定程度上约束了你的思维，甚至你忽略最原始的编程思想也能顺利完成。

所以大家在网上还是尽量避开用 Vue 还是 React, 用 JS 还是 TS 的争论。因为你如果不是一个公司的员工，那你用什么都没有人限制呢，如果你是一个普通员工，那用什么其实和你没关系，这是上层或是公司决定的事。而真正的掌控者/受益者肯定不会纠结于技术到底怎么用。

所以大家对于选择什么框架，使用什么生态，都不要过分纠结。要学习的是整个流程中对产品的设计思想。框架和语言只是一个熟练度问题，写多了都一样的。
